# Commit

변경사항을 분석하고 Conventional Commits 형식으로 커밋을 생성합니다.

## 단계

1. `git status`로 변경사항 확인
2. `git diff --staged`로 스테이징된 변경사항 분석
3. 스테이징된 파일이 없으면 `git diff`로 전체 변경사항 확인
4. **변경사항을 논리적 작업 단위로 분리** (기능 추가, 버그 수정, 리팩토링 등 성격이 다른 변경은 별도 커밋)
5. 각 단위별로 파일을 스테이징하고 커밋 생성
6. 모든 단위가 커밋될 때까지 반복

---

## 커밋 메시지 형식

```
{타입}: {제목}

{본문 (선택)}

Co-Authored-By: Claude <noreply@anthropic.com>
```

## 타입

| 타입       | 설명        | 예시                          |
| ---------- | ----------- | ----------------------------- |
| `feat`     | 새로운 기능 | `feat: 로그인 기능 추가`      |
| `fix`      | 버그 수정   | `fix: null pointer 에러 수정` |
| `refactor` | 리팩토링    | `refactor: 유틸 함수 정리`    |
| `docs`     | 문서 수정   | `docs: README 업데이트`       |
| `chore`    | 기타 작업   | `chore: 의존성 업데이트`      |
| `test`     | 테스트      | `test: 유닛 테스트 추가`      |
| `enhance`  | 기능 개선   | `enhance: 성능 최적화`        |

## 규칙

### 제목

- 타입은 **소문자**로 작성
- 타입 뒤에 **콜론과 공백** 추가
- 설명은 **명령형**으로 간결하게 작성
- 한글/영문 모두 가능

### 본문 (선택)

- 제목과 **빈 줄**로 구분
- **왜** 변경했는지 설명
- 변경의 **배경**이나 **맥락** 기술

---

## 실행

위 규칙에 따라 변경사항을 분석하고 커밋을 생성합니다.
