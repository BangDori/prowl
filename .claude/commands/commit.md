# Commit

변경사항을 분석하고 Conventional Commits 형식으로 커밋을 생성합니다.

## 단계

1. `git status`로 변경사항 확인
2. `git diff --staged`로 스테이징된 변경사항 분석
3. 스테이징된 파일이 없으면 `git diff`로 전체 변경사항 확인
4. **변경사항을 논리적 작업 단위로 분리** (기능 추가, 버그 수정, 리팩토링 등 성격이 다른 변경은 별도 커밋)
5. 각 단위별로 파일을 스테이징하고 커밋 생성
6. 모든 단위가 커밋될 때까지 반복

---

## 커밋 메시지 형식

```
{타입}: {제목}

{본문 (선택)}

Co-Authored-By: Claude <noreply@anthropic.com>
```

## 타입 (소문자, 영어)

| 타입       | 설명                                    | 예시                                     |
| ---------- | --------------------------------------- | ---------------------------------------- |
| `feat`     | 새로운 기능 추가                        | `feat: 스플래시 화면 눈 뜨기 애니메이션 추가` |
| `fix`      | 버그 수정                               | `fix: 새벽 작업 알림 재전송 간격 1분으로 수정` |
| `refactor` | 코드 리팩토링 (동작 변경 없음)          | `refactor: launchd 서비스 타입 정리`     |
| `style`    | 포맷팅, 세미콜론 등 코드 의미 변경 없음 | `style: 불필요한 공백 제거`              |
| `docs`     | 문서 변경                               | `docs: README에 아키텍처 다이어그램 추가` |
| `test`     | 테스트 추가/수정                        | `test: IPC 핸들러 단위 테스트 추가`      |
| `chore`    | 빌드, 설정, 패키지 등 기타 변경         | `chore: postinstall 스크립트에 electron 설치 추가` |
| `perf`     | 성능 개선                               | `perf: 로그 파싱 속도 최적화`            |
| `ci`       | CI/CD 설정 변경                         | `ci: PR 대상 CI 워크플로우 추가`         |

## 규칙

### 제목

- 타입은 **소문자**로 작성
- 타입 뒤에 **콜론과 공백** 추가 (`feat: `)
- 설명은 **한국어 명령형**으로 간결하게 작성
- 끝에 마침표 없음
- 50자 이내

### 본문 (선택)

- 제목과 **빈 줄**로 구분
- **왜** 변경했는지 설명
- 변경의 **배경**이나 **맥락** 기술

---

## 실행

위 규칙에 따라 변경사항을 분석하고 커밋을 생성합니다.
